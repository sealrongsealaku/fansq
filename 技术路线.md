# 互联网支教教学反思墙系统技术路线说明

## 一、技术路线目标

一期技术路线遵循四个原则：

1. 够用，不做过度设计
2. 稳定，优先成熟方案
3. 易维护，便于校内服务器长期运行
4. 易扩展，为后续展示优化和功能扩展留空间

本项目一期不是高并发平台，也不是复杂分布式系统，重点是稳定打通：

**QQ 群 @机器人提交 -> 后端入库 -> 后台人工审核 -> 前台展示**

所以整体采用 **单体业务服务 + 独立机器人服务 + 前端分离部署** 的方案。

## 二、推荐技术选型

## 2.1 前端

### 管理后台

- `Vue 3`
- `Vite`
- `TypeScript`
- `Element Plus`
- `Vue Router`
- `Pinia`
- `Axios`

选择原因：

1. 后台表格、筛选、表单、弹窗场景多，`Element Plus` 很适合快速交付。
2. `Vue 3 + Vite` 启动快、构建快、学习成本低。
3. `TypeScript` 有助于接口字段和状态管理保持一致。

### 前台反思墙

- `Vue 3`
- `Vite`
- `TypeScript`
- 自定义 CSS / SCSS

选择原因：

1. 前台更强调设计感和阅读感，不适合直接套后台组件库风格。
2. 使用同一前端技术栈，能降低维护成本。
3. 后续做大屏展示、点赞交互和动效也更灵活。

## 2.2 后端

- `Node.js`
- `NestJS`
- `TypeScript`
- `Prisma` 或 `TypeORM`
- `JWT` 或服务端 Session
- `class-validator`

选择原因：

1. `NestJS` 分层清晰，适合做中小型但需要规范结构的业务系统。
2. 控制器、服务、模块的边界明确，后续维护舒服。
3. 与前端、机器人统一使用 `TypeScript`，上下游协作更顺。
4. 一期接口以 CRUD 和审核流转为主，Nest 非常合适。

关于 ORM：

- 如果想要更现代、迁移体验更顺，优先 `Prisma`
- 如果更习惯传统实体映射，可以选 `TypeORM`

我的建议是一期开 `Prisma`，因为：

1. 数据模型直观
2. 迁移方便
3. 类型推导体验更好

## 2.3 数据库

- `MySQL 8.0`

选择原因：

1. 部署普遍，校内环境常见
2. 运维资料多，维护成本低
3. 一期查询场景简单，MySQL 足够

## 2.4 QQ 机器人服务

- `Node.js`
- `TypeScript`
- WebSocket 客户端
- `Axios` / `fetch` 调用后端内部接口

选择原因：

1. 机器人本质是接入层，不需要复杂框架
2. 保持轻量，更容易稳定运行和排障
3. 与主业务后端解耦，后续替换接入方式成本低

## 2.5 部署与运维

- `Ubuntu`
- `Nginx`
- `systemd`
- `PM2` 可选，但优先 `systemd`

选择原因：

1. 校内服务器环境适配度高
2. `Nginx` 适合做静态资源托管和反向代理
3. `systemd` 对常驻服务守护和开机自启更直接

## 三、总体架构

推荐采用四层结构：

1. 前台展示前端
2. 后台管理前端
3. 后端业务服务
4. QQ 机器人接入服务

底层统一使用 MySQL 存储。

## 3.1 架构关系

```text
QQ 群消息
   ->
QQ 机器人服务
   ->
后端内部提交接口
   ->
MySQL
   ->
后台管理接口 / 前台公开接口
   ->
后台前端 / 前台反思墙
```

## 3.2 模块职责

### 1. 前台反思墙

负责：

1. 展示审核通过且允许展示的内容
2. 提供普通浏览模式和大屏模式
3. 承担项目展示和传播作用

不负责：

1. 审核逻辑
2. 机器人接入逻辑
3. 复杂业务处理

### 2. 后台管理端

负责：

1. 管理员登录
2. 反思列表查看
3. 内容审核、编辑、展示控制
4. 基础统计展示

### 3. 后端业务服务

负责：

1. 身份认证
2. 数据入库与查询
3. 审核状态流转
4. 前台公开接口
5. 后台管理接口
6. 审计日志

### 4. QQ 机器人服务

负责：

1. 连接 QQ 平台 WebSocket
2. 监听群内 `@机器人` 消息
3. 提取正文和基础信息
4. 调用后端内部接口入库
5. 回复提示信息
6. 断线重连

## 四、为什么不建议更重的方案

一期不建议直接上以下方案：

1. 微服务
2. 消息队列
3. Elasticsearch
4. Redis 必选化
5. 容器编排

原因不是这些技术不好，而是当前项目没有必要。

这个系统的一期核心问题不是高并发，也不是海量全文检索，而是：

1. 内容采集稳定
2. 审核流程清晰
3. 前台展示有品质
4. 部署简单可维护

如果一开始把架构做得太重，维护成本会明显高于业务收益。

## 六、环境划分建议

建议最少区分两个环境：

1. `development`
2. `production`

如果条件允许，可加：

3. `staging`

环境变量建议按服务拆分：

### 后端

- `PORT`
- `DATABASE_URL`
- `JWT_SECRET`
- `ADMIN_INIT_USERNAME`
- `ADMIN_INIT_PASSWORD_HASH`
- `INTERNAL_API_TOKEN`

### 机器人

- `QQ_BOT_APP_ID`
- `QQ_BOT_TOKEN`
- `QQ_BOT_SECRET`
- `QQ_WS_URL`
- `API_BASE_URL`
- `INTERNAL_API_TOKEN`

### 前端

- `VITE_API_BASE_URL`

## 七、部署方案

## 7.1 推荐部署方式

在一台 Ubuntu 服务器上部署以下内容：

1. `Nginx`
2. `admin-web` 静态文件
3. `public-web` 静态文件
4. `api-server` 进程
5. `qq-bot` 进程
6. `MySQL`

## 7.2 Nginx 分工

1. 提供前台静态页面
2. 提供后台静态页面
3. 将 `/api` 反向代理到后端服务

## 7.3 systemd 分工

1. 守护 `api-server`
2. 守护 `qq-bot`
3. 支持开机自启
4. 支持故障自动重启

## 八、关键实现建议

## 8.1 后端设计建议

1. 机器人提交接口与后台接口分开
2. 前台公开接口不要直接暴露后台字段
3. 审核操作统一写入审计日志
4. 匿名显示逻辑由后端处理，不交给前端判断
5. 点赞去重由数据库唯一约束兜底，不只依赖前端按钮状态

## 8.2 前端设计建议

1. 后台和前台使用不同布局和视觉风格
2. 后台优先效率，前台优先阅读体验
3. 前台设计不要做成普通新闻列表
4. 点赞状态建议通过本地 `visitor_id` 加接口查询统一管理

## 8.3 机器人设计建议

1. 严格只处理 `@机器人` 的群消息
2. 消息解析失败要记录日志
3. 调用后端失败时要有重试或补偿策略
4. 自动回复内容尽量简短稳定

## 九、我对一期的最终建议

一期最稳妥的技术路线是：

1. `Vue 3 + Vite + TypeScript` 做前端
2. `Element Plus` 仅用于后台
3. `NestJS + Prisma + MySQL` 做后端
4. `Node.js + TypeScript + WebSocket` 做 QQ 机器人服务
5. `MySQL` 中增加点赞记录表和唯一索引用于点赞去重
6. `Nginx + systemd + Ubuntu` 完成部署

这条路线的优点是：

1. 开发效率高
2. 架构不重
3. 部署和维护成本低
4. 后续扩展空间足够

## 十、下一步建议

基于这条技术路线，下一步最适合直接做的是：

1. 初始化项目目录结构
2. 搭建 Monorepo 基础骨架
3. 创建前端、后端、机器人三个应用
4. 接入数据库和基础登录能力
