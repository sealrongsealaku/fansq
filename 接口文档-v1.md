# 互联网支教教学反思墙系统接口文档 v1

## 一、约定说明

## 1.1 基础约定

1. 所有接口默认使用 `JSON` 进行请求与响应。
2. 时间字段统一使用 ISO 8601 格式，例如：`2026-04-11T15:30:00+08:00`。
3. 响应统一包含 `success`、`message`、`data` 字段。

标准响应示例：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

失败响应示例：

```json
{
  "success": false,
  "message": "unauthorized",
  "data": null
}
```

## 1.2 状态枚举

### 审核状态 `audit_status`

- `pending`：待审核
- `approved`：审核通过
- `rejected`：审核驳回

### 展示状态 `display_status`

- `hidden`：隐藏
- `visible`：展示中

## 1.3 认证方式

### 管理员后台接口

采用 Bearer Token 方式。

请求头示例：

```http
Authorization: Bearer <access_token>
```

### 机器人内部接口

采用内部 Token 校验。

请求头示例：

```http
X-Internal-Token: <internal_token>
```

### 前台点赞身份识别

一期点赞不要求登录。

前台在浏览器首次访问时生成一个 `visitor_id`，持久化保存在本地，并在需要点赞或查询点赞状态时通过请求头传递：

```http
X-Visitor-Id: <visitor_id>
```

该标识用于限制同一用户对同一条反思重复点赞。

## 二、机器人接入接口

## 2.1 提交反思

### `POST /api/internal/reflections/qq-submit`

用途：
供 QQ 机器人服务调用，将群消息写入业务系统。

请求体：

```json
{
  "student_name": "小雨同学",
  "submit_content": "今天的课堂我学会了分数比较，老师讲得很清楚。",
  "submit_time": "2026-04-11T15:30:00+08:00",
  "submit_channel": "qq_group_bot",
  "source_group_id": "123456789",
  "source_group_name": "支教项目六年级群",
  "raw_message_id": "msg_001"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| student_name | string | 是 | QQ 群昵称 |
| submit_content | string | 是 | 反思正文 |
| submit_time | string | 是 | 提交时间 |
| submit_channel | string | 是 | 固定为 `qq_group_bot` |
| source_group_id | string | 否 | QQ 群标识 |
| source_group_name | string | 否 | QQ 群名称 |
| raw_message_id | string | 否 | 原始消息 ID |

成功响应：

```json
{
  "success": true,
  "message": "received",
  "data": {
    "id": 1001,
    "audit_status": "pending",
    "display_status": "hidden"
  }
}
```

失败场景：

1. `400`：正文为空或格式不合法
2. `401`：内部 Token 无效
3. `500`：服务端入库失败

## 4.3 修改反思内容

### `PATCH /api/admin/reflections/:id`

请求体：

```json
{
  "submit_content": "今天的课堂我学会了分数比较，也更愿意主动发言。",
  "display_status": "hidden",
  "is_featured": true,
  "is_top": false,
  "is_anonymous": true,
  "display_name": "一位同学",
  "remarks": "表达清晰，适合精选"
}
```

可修改字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| submit_content | string | 否 | 修改正文 |
| display_status | string | 否 | `hidden/visible` |
| is_featured | boolean | 否 | 是否精选 |
| is_top | boolean | 否 | 是否置顶 |
| is_anonymous | boolean | 否 | 是否匿名展示 |
| display_name | string | 否 | 匿名或展示名称 |
| remarks | string | 否 | 管理员备注 |

成功响应：

```json
{
  "success": true,
  "message": "updated",
  "data": {
    "id": 1001
  }
}
```

## 4.4 审核通过

### `POST /api/admin/reflections/:id/approve`

请求体：

```json
{
  "display_status": "visible"
}
```

说明：

1. 审核通过后 `audit_status` 更新为 `approved`
2. 可由管理员决定是否立即展示

成功响应：

```json
{
  "success": true,
  "message": "approved",
  "data": {
    "id": 1001,
    "audit_status": "approved",
    "display_status": "visible"
  }
}
```

## 4.5 审核驳回

### `POST /api/admin/reflections/:id/reject`

请求体：

```json
{
  "remarks": "内容过短，请补充后再提交"
}
```

说明：

1. 驳回后 `audit_status` 更新为 `rejected`
2. `display_status` 强制为 `hidden`

成功响应：

```json
{
  "success": true,
  "message": "rejected",
  "data": {
    "id": 1001,
    "audit_status": "rejected",
    "display_status": "hidden"
  }
}
```

## 4.6 控制展示状态

### `POST /api/admin/reflections/:id/visibility`

请求体：

```json
{
  "display_status": "visible"
}
```

规则：

1. 只有 `approved` 的内容才能设置为 `visible`
2. `pending` 或 `rejected` 内容只能保持 `hidden`

成功响应：

```json
{
  "success": true,
  "message": "visibility_updated",
  "data": {
    "id": 1001,
    "display_status": "visible"
  }
}
```

失败场景：

1. `400`：未审核通过的内容不能公开展示

## 五、前台公开接口

## 5.1 获取前台反思列表

### `GET /api/public/reflections`

查询参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| page | number | 否 | 页码，默认 1 |
| page_size | number | 否 | 每页数量，默认 20 |
| featured_only | boolean | 否 | 是否只看精选 |

返回规则：

1. 仅返回 `audit_status = approved` 且 `display_status = visible` 的记录
2. 返回给前台的 `display_name` 应为最终可展示名称
3. 默认排序为 `is_top desc, submit_time desc`
4. 若请求头包含 `X-Visitor-Id`，可同时返回当前用户是否已点赞的 `liked` 字段

成功响应：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "list": [
      {
        "id": 1001,
        "display_name": "一位同学",
        "like_count": 12,
        "submit_content": "今天的课堂我学会了分数比较，也更愿意主动发言。",
        "submit_time": "2026-04-11T15:30:00+08:00",
        "source_group_name": "支教项目六年级群",

}
```

## 5.3 点赞反思

### `POST /api/public/reflections/:id/like`

请求头：

```http
X-Visitor-Id: <visitor_id>
```

规则：

1. 仅允许对 `approved + visible` 的内容点赞
2. 同一 `visitor_id` 对同一条反思只能点赞一次
3. 点赞成功后写入点赞记录，并将 `like_count + 1`

成功响应：

```json
{
  "success": true,
  "message": "liked",
  "data": {
    "id": 1001,
    "like_count": 13,
    "liked": true
  }
}
```

失败场景：

1. `400`：缺少 `X-Visitor-Id`
2. `404`：反思不存在或未公开展示
3. `409`：当前用户已点赞，不能重复点赞

## 5.4 获取点赞状态

### `GET /api/public/reflections/:id/like-status`

请求头：

```http
X-Visitor-Id: <visitor_id>
```

## 六、错误码建议

| 状态码 | 含义 | 说明 |
| --- | --- | --- |
| 200 | 成功 | 请求成功 |
| 400 | 参数错误 | 请求参数不合法 |
| 401 | 未认证 | Token 缺失或无效 |
| 403 | 无权限 | 权限不足 |
| 404 | 未找到 | 资源不存在 |
| 409 | 状态冲突 | 例如未审核通过却尝试公开展示 |
| 500 | 服务异常 | 服务端错误 |

## 七、开发实现建议

1. 后台列表和详情接口统一返回原始字段，便于管理操作。
2. 前台公开接口返回已加工后的显示名，不要让前端自行判断匿名逻辑。
3. 审核、展示、编辑操作均写入 `audit_log`。
4. 机器人提交接口与后台接口分离，避免权限混乱。
5. 点赞接口用唯一约束保证不可重复点赞，不要只依赖前端禁用按钮。
6. 后续如果增加图片或教师点评，优先扩展字段和子表，不要改动当前主链路接口语义。
