# 互联网支教教学反思墙系统一期拆解文档

## 一、拆解目标

基于 [需求.md](/D:/FanSQ/需求.md) 的一期范围，将系统进一步拆解为可直接进入设计、开发和联调的实现清单。

一期只围绕以下主链路展开：

**QQ 群 @机器人提交 -> 机器人服务接收 -> 后端入库 -> 管理员人工审核 -> 前台展示**

## 二、一期功能边界

### 1. 做什么

1. 学生通过 QQ 群 `@机器人` 提交反思。
2. 机器人服务接收消息并调用后端接口入库。
3. 后台管理员登录后对内容进行审核、编辑、发布管理。
4. 前台展示审核通过的内容。
5. 前台支持轻量点赞，且同一用户不可重复点赞。
6. 支持基础统计和大屏展示。

### 2. 不做什么

1. 不做网页表单提交。
2. 不做微信提交。
3. 不做评论、排行。
4. 不做敏感词检测和自动审核。
5. 不做图片、音频、视频上传。
6. 不做复杂数据分析。

## 三、页面拆解

## 3.1 前台页面

### 页面 1：反思墙主页

用途：
展示审核通过且允许展示的反思内容，适用于电脑端和移动端日常浏览。

核心区块：

1. 顶部引导区
   - 项目名称
   - 页面标题
   - 简短说明
2. 内容展示区
   - 精选反思区
   - 普通反思列表区
   - 排序说明或时间说明
3. 底部说明区
   - 项目介绍
   - 时间范围
   - 主办单位
   - 二维码或说明信息

核心交互：

1. 支持分页或懒加载。
2. 支持按默认规则排序。
3. 支持匿名名称展示。
4. 支持点赞与点赞数展示。

### 页面 2：大屏展示页

用途：
用于活动现场、会议室、展厅等大屏场景。

核心特点：

1. 默认全屏展示。
2. 自动滚动或轮播。
3. 字体更大，信息更简洁。
4. 优先展示精选或置顶内容。

## 四、角色与权限拆解

### 1. 学生

权限：

1. 仅能在 QQ 群通过 `@机器人` 提交内容。
2. 不直接访问后台。
3. 不参与审核操作。

### 2. 管理员

权限：

1. 登录后台。
2. 查看全部反思内容。
3. 审核、编辑、删除、展示控制。
4. 查看基础统计。

### 3. 公共浏览用户

权限：

1. 访问前台公开展示页。
2. 仅能看到审核通过且允许展示的内容。
3. 可对已展示内容点赞，但同一用户不能对同一条内容重复点赞。

## 五、核心业务规则

### 1. 提交规则

1. 只有群内 `@机器人` 的消息才作为反思提交处理。
2. 空内容或只有 `@` 无正文的消息不入库。
3. 机器人收到有效内容后可回复“已收到反思”。

### 2. 审核规则

1. 所有新提交内容默认 `待审核`。
2. 只有管理员可将内容改为 `审核通过`。
3. 审核未通过的内容不得出现在前台。

### 3. 展示规则

1. 前台展示条件必须同时满足：
   - `audit_status = approved`
   - `display_status = visible`
2. 置顶内容优先展示。
3. 精选内容可单独做突出展示区域。
4. 若 `is_anonymous = true`，前台展示 `display_name`；否则展示 `student_name`。

### 4. 排序规则

前台默认排序建议：

1. `is_top` 降序
2. `submit_time` 降序

若存在精选专区：

1. 精选区按 `is_top` 降序、`submit_time` 降序
2. 普通区展示非精选内容，按 `submit_time` 降序

### 5. 点赞规则

1. 只有前台公开展示的内容允许点赞。
2. 同一用户对同一条反思只能点赞一次。
3. 点赞成功后累计点赞数加 1。
4. 重复点赞时接口应明确返回“已点赞”。
5. 一期点赞不要求登录。
6. 一期采用浏览器侧唯一标识配合服务端记录的方式限制重复点赞。

## 六、状态设计

## 6.1 审核状态 audit_status

建议枚举：

1. `pending`：待审核
2. `approved`：审核通过
3. `rejected`：审核驳回

## 6.2 展示状态 display_status

建议枚举：

1. `hidden`：不展示
2. `visible`：展示中

## 6.3 推荐状态流转

新内容入库：

`pending + hidden`

管理员审核通过并允许展示：

`approved + visible`

管理员审核驳回：

`rejected + hidden`

管理员审核通过但暂不展示：

`approved + hidden`

## 八、接口拆解

## 8.1 机器人接入接口

### `POST /api/internal/reflections/qq-submit`

用途：
供 QQ 机器人服务调用，将群消息写入业务系统。

请求示例：

```json
{
  "student_name": "小雨同学",
  "submit_content": "今天的课堂我学会了分数比较，老师讲得很清楚。",
  "submit_time": "2026-04-11T15:30:00+08:00",
  "submit_channel": "qq_group_bot",
  "source_group_id": "123456789",
  "source_group_name": "支教项目六年级群",
  "raw_message_id": "msg_001"
}
```

响应示例：

```json
{
  "success": true,
  "message": "received",
  "data": {
    "id": 1001,
    "audit_status": "pending"
  }
}
```

接口规则：

1. 仅允许内部服务调用。
2. 通过内部 Token 或签名校验。
3. 成功入库后默认状态为 `pending + hidden`。

## 8.2 管理员认证接口

### `POST /api/admin/login`

用途：
管理员登录。

请求字段：

```json
{
  "username": "admin",
  "password": "******"
}
```

### `POST /api/admin/logout`

用途：
管理员退出登录。

### `GET /api/admin/me`

用途：
获取当前登录管理员信息。

## 8.3 反思管理接口

### `GET /api/admin/reflections`

用途：
分页获取反思列表。

支持参数：

1. `page`
2. `page_size`
3. `keyword`
4. `audit_status`
5. `display_status`
6. `source_group_id`
7. `start_time`
8. `end_time`

### `GET /api/admin/reflections/:id`

用途：
获取单条反思详情。

### `PATCH /api/admin/reflections/:id`

用途：
修改反思内容和展示属性。

可修改字段建议：

1. `submit_content`
2. `display_status`
3. `is_featured`
4. `is_top`
5. `is_anonymous`
6. `display_name`
7. `remarks`

### `POST /api/admin/reflections/:id/approve`

用途：
审核通过。

建议行为：

1. 将 `audit_status` 更新为 `approved`
2. 可同时决定是否直接 `visible`

### `POST /api/admin/reflections/:id/reject`

用途：
审核驳回。

建议行为：

1. 将 `audit_status` 更新为 `rejected`
2. `display_status` 置为 `hidden`

### `POST /api/admin/reflections/:id/visibility`

用途：
控制展示 / 隐藏。

请求示例：

```json
{
  "display_status": "visible"
}
```

### `DELETE /api/admin/reflections/:id`

用途：
删除反思。

建议：

1. 若业务允许，优先采用软删除。
2. 若采用硬删除，需保留审计日志。

## 8.4 前台展示接口

### `GET /api/public/reflections`

用途：
获取前台展示用反思列表。

返回规则：

1. 只返回 `approved + visible` 的数据。
2. 前台若匿名展示，只返回处理后的显示名。
3. 默认按 `is_top desc, submit_time desc` 排序。

建议支持参数：

1. `page`
2. `page_size`
3. `featured_only`

### `POST /api/public/reflections/:id/like`

用途：
对公开展示的反思进行点赞。

接口规则：

1. 仅允许对 `approved + visible` 的内容点赞。
2. 请求需携带前台访问者唯一标识。
3. 同一 `visitor_id` 对同一条反思只能点赞一次。
4. 点赞成功后写入 `reflection_like`，并更新 `like_count`。

### `GET /api/public/reflections/:id/like-status`

用途：
查询当前访问者是否已对该反思点赞。

适用场景：

1. 页面初始化回显点赞状态。
2. 点赞按钮禁用状态判断。

### `GET /api/public/summary`

用途：
获取前台展示页顶部或底部的统计和说明数据。

返回建议包含：

1. 项目名称
2. 时间范围
3. 总展示数量
4. 精选数量

## 九、机器人服务拆解

机器人服务职责建议保持最小化：

1. 连接 QQ 平台 WebSocket。
2. 监听群消息事件。
3. 判断是否为 `@机器人` 消息。
4. 解析反思正文。
5. 调用后端内部接口提交数据。
6. 向群内回复成功或失败提示。
7. 断线后自动重连。

不建议机器人服务承担：

1. 审核逻辑
2. 前台展示逻辑
3. 后台管理逻辑
4. 直接面向用户的复杂业务判断

## 十、前端展示规则拆解

## 10.1 反思卡片字段建议

卡片可展示：

1. 显示名称
2. 反思正文
3. 提交时间
4. 来源群名称
5. 精选标识
6. 点赞数

卡片不建议展示：

1. 原始消息 ID
2. 管理员备注
3. 审核状态

## 10.2 文案处理建议

1. 长文本可折叠或限制行数后展开。
2. 时间建议格式化为年月日时分。
3. 匿名时统一使用 `display_name`，为空时可降级为“匿名同学”。
4. 点赞成功后建议即时刷新卡片点赞数。

## 10.3 大屏模式规则

1. 默认隐藏筛选和复杂操作。
2. 保持自动滚动。
3. 放大字号与卡片间距。
4. 精选内容可优先轮播。

## 十一、统计口径拆解

一期建议只做简单、稳定的统计：

1. 总提交数
   - 全部已入库反思总量
2. 待审核数
   - `audit_status = pending`
3. 已通过数
   - `audit_status = approved`
4. 已展示数
   - `approved + visible`
5. 精选数
   - `is_featured = true`
6. 点赞总数
   - 所有反思 `like_count` 汇总
7. 群聊提交数
   - 按 `source_group_id` 聚合统计
8. 每日 / 每周趋势
   - 按 `submit_time` 聚合统计

## 十四、验收标准建议

### 1. 提交链路

1. 学生在群内 `@机器人` 提交后，内容能成功入库。
2. 无正文消息不会误入库。
3. 机器人异常断线后可自动恢复。

### 2. 审核链路

1. 新内容默认处于待审核状态。
2. 管理员可完成通过、驳回、编辑、隐藏、置顶、精选操作。
3. 后台修改后前台展示结果正确。

### 3. 展示链路

1. 未审核内容不会出现在前台。
2. 匿名内容展示匿名名称。
3. 大屏页可正常自动播放或滚动。
4. 同一用户对同一条内容重复点赞时不会重复计数。

### 4. 部署链路

1. 系统在 Ubuntu 服务器上可稳定运行。
2. 前台、后台、机器人服务均可独立启动。
3. 重启服务器后服务可自动拉起。
